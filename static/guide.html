這是什麼

HiSKIO AI 客服系統是一個三段式對話客服雛形： 先由 主管（Sonnet） 看完整對話脈絡決定要做什麼動作，再分派給對應的執行者節點（FAQ 回應、KB 查詢、離題擋下、建立工單等）回應用戶。

最終目標：用戶能透過對話得到 FAQ 標準答案、KB 知識庫解答；遇到複雜或非業務範圍的問題會自動建議建立工單由真人客服處理。

內部操作者主要看左側對話、右側即時狀態（State 除錯面板），可隨時了解 AI 為什麼這樣回應、走哪一條路徑。

對話訊息燈號說明

每則 AI 回應左側的色帶代表系統判斷的回應類型，可從顏色快速看出是走哪一條路徑：

淺藍虛線 — 問候 (greeting)純打招呼（你好、Hi、在嗎）。turn_count 不增加。一個 session 累計 max_greeting_count 次（預設 3）後永久轉灰框。
灰色 — 問候已達上限 (greeting_blocked)greeting 累計超過 3 次後的硬擋訊息，提示用戶有問題就直接描述、否則請結束對話。不再呼叫 LLM。
橘黃 — 釐清 (clarification)用戶訊息模糊（亂碼、太籠統、無法判斷想問什麼），系統請用戶補充。連續 3 次強制建單。
橘色 — 離題 (off_topic)用戶問題不在 HiSKIO 業務範圍（食物、天氣、推薦商品、質疑系統等）。off_topic_count 會累加，達上限後觸發建單建議。
灰色（小字）— 離題已達上限 (off_topic_blocked)離題累計達上限後的硬擋訊息：「對話僅處理 HiSKIO 服務相關問題」。
深紫色 — 多重意圖選項 (intent_selection)偵測到多個獨立問題，列選項請用戶選一個先處理。
綠色 — FAQ 標準答案 (faq)命中 FAQ，AI 用核心步驟一字不漏照抄、潤飾開場結尾。穩定、不會幻覺。
藍色 — 知識庫解答 (rag)FAQ 未命中、KB 有相關文章，由 Sonnet 根據 KB 內容生成回答。
粉紅色 — 知識庫無解 (no_kb_match)KB 完全沒有相關文章，AI 誠實告知並建議建立工單。
紫色 — 工單流程 (ticket_flow / force_escalation)建議建單／確認建單／收 Email／建單成功 等工單相關訊息。force_escalation 是「連續 3 次無法溝通」的強制建單。
草綠 — 確認語回應 (acknowledge) ⭐ v7 新增用戶說「謝謝」「我知道了」「OK」等確認語，主管直接回禮貌過渡語並把當前 intent 標 confirmed_resolved，不再走 KB（解 v6 鬼打牆）。
紅色 — 對話結束 (session_ended)工單已建立或對話結束，提示用戶按「新對話」重新開始。

主管 Action 說明（v6 / v7 共用）

v6 開始用 Sonnet 主管統一決策，每輪訊息進來主管從以下 12 種 action 選 1。v5 沒有 action enum（用流水線分支分流），v5 的對應路徑會在「系統運作流程」分頁的 v5 tab 標出。

action何時用備註
answer_with_faq命中 FAQ，用標準答案最常見
answer_with_kbFAQ 未命中但 KB 有相關文章常見
clarify用戶訊息模糊，需澄清常見
continue_intent用戶在補充當前意圖細節常見
list_pending_intents用戶用「下一個」這類指稱詞，列出待辦意圖多重意圖場景
acknowledge_out_of_scope不在 HiSKIO 業務範圍離題訊息
acknowledge_uncertainty主管不確定用戶想問什麼，誠實告知邊界 case（v6 引入）
acknowledge_confirmation ⭐用戶說「謝謝 / 我知道了 / OK」等確認語v7 新增。回應由 acknowledge_handler（Haiku）寫，自動推進到下一個 pending intent
suggest_ticket建議建立工單需要人工介入時
force_escalation連續多次無法溝通，強制建單很少觸發
greeting純打招呼備案，多數情況被 fast-path 先攔
常見誤解澄清：「greeting 是不是用 Sonnet 處理？」
  • 純招呼（regex 命中）→ 完全不打 Sonnet，由 Haiku 直接生問候
  • 帶錯字／變化型招呼（regex 沒命中）→ Sonnet 主管接手判斷後也選 greeting action → 還是叫 Haiku 生問候
  • 無論哪條路徑，最終問候訊息都是 Haiku 生成的。差別只在「有沒有呼叫 Sonnet 做路由判斷」（影響成本）

系統運作流程（v5 / v6 / v7 切換）

每個版本的架構不同。下方分頁切換對照各版本的：簡述 / 架構圖 / 解決什麼問題 / 踩到什麼新 bug / 實測成本。

v5 — 分類器流水線（已歸檔，branch=v5-pipeline-mode）

核心想法：4 個 Haiku 接力跑，每節點看自己負責的一小塊資料。沒有「主管」、沒有 action enum。

[用戶訊息] ↓ ┌────────────────────────────────┐ │ entry_classifier（Haiku） │ → greeting / unclear / off_topic / customer_service └────────────────────────────────┘ ↓ customer_service ┌────────────────────────────────┐ │ intent_clarity（Haiku） │ → simple / parallel / 指稱詞 └────────────────────────────────┘ ↓ ┌────────────────────────────────┐ │ faq_matcher（Haiku） │ → matched_id + confidence └────────────────────────────────┘ ↓ confidence ≥ 0.7 ↓ confidence < 0.7 [faq_responder Haiku] [kb_indexer Haiku] ↓ [cs_response Sonnet] ↓ [evaluator Haiku 背景] → 寫筆記到 state

解的問題（相比更早 v3）

  • 把入口分類拆細（greeting / unclear / off_topic / customer_service）
  • 加意圖明確度節點（處理多重意圖、指稱詞）
  • 加 intent_log 結構追蹤所有意圖 status

踩到的 bug

bug實況
「人工建單為什麼以後不能動作了」鬼打牆 ⭐entry_classifier 看到「人工建單」字眼歸 customer_service → kb_indexer 被歷史污染挑 kb_005（發票）→ cs_response 拿到 KB 全文回「請問您的發票問題是哪方面...」。沒有任何節點看出用戶在質疑系統。
同樣意思走不同 path 結果不一致選號「3」走 phase 攔截、陳述句「我有水果問題」走 entry_classifier，兩條路結果不一樣。
ticket_state.declined 永久封印用戶拒絕一次工單後、之後新意圖也不再被建議建單。
邊界 case 永遠補不完新訊號要在 entry_classifier、faq_matcher、kb_indexer、cs_response 都加分支。

v6 — 主管統一決策（已歸檔，branch=v6-manager-mode）

核心想法：拆掉 4 個 Haiku 分類器，改由一個 Sonnet 主管看「全貌」一次決策，輸出 11 個 action 之一，分派給 executor 節點生回應。

[用戶訊息] ↓ ┌────────────────────────────────┐ │ phase 攔截 / greeting fast-path │ 純程式 / regex └────────────────────────────────┘ ↓ 沒命中 ┌────────────────────────────────────────────┐ │ manager.decide（Sonnet） │ │ 看：用戶這句 + 完整 history（10 輪） │ │ + intent_log + 服務狀態 │ │ + 完整 FAQ 全表 + 完整 KB 索引 │ │ 輸出：recommended_action（11 選 1）+ payload│ └────────────────────────────────────────────┘ ↓ ┌────────────────────────────────┐ │ orchestrator._execute_* │ 純程式派工 └────────────────────────────────┘ ↓ ┌─ executor 節點 ────────────────┐ │ faq_responder Haiku │ │ cs_response Sonnet (cached) │ │ no_kb_handler Haiku │ │ off_topic Haiku │ │ ticket_handler Haiku/Sonnet │ │ ... │ └────────────────────────────────┘ ↓ [evaluator Haiku 背景]

解的問題（v5 → v6）

  • v5「人工建單」鬼打牆：主管看完整 history 能識別 meta 質疑 → 選 suggest_ticket
  • 路徑不一致：全部進主管，只有一條 code path
  • 「我聽不懂」一級公民：新增 acknowledge_uncertainty action
  • 邊界 case 加 action enum 一處改完，不必到處改節點

v6 自己踩到的 bug（驅動 v7 升級）

bug實況
「OK 了 謝謝」鬼打牆 ⭐主管的 11 個 action 沒對應 confirmation 出口、硬塞 continue_intent → 把當前 intent 主題又重答一遍。
evaluator 漏判 confirmed_resolvedprompt「對齊 5 個範例」太死板、Haiku 在 t=0 嚴格字面對齊，「我知道了 / 了解了」這類常見表達被判 false。
主管 prompt 越加越笨FAQ 全表 + KB 索引 + 規則都壓在 system prompt、Sonnet 注意力分散、混合意圖（in/out scope）容易判 clarify。

成本

NT$3.81 / 10 輪（開 Sonnet prompt cache 後實測）

v7.2 — 流水線權威 + 主管最終決策 + Haiku acknowledge handler（current main）

核心想法：把 v5 流水線復活做為「FAQ/KB 比對權威」，主管 Sonnet 改吃流水線「提議卡」，從 12 個 action 選 1（多了 acknowledge_confirmation）。主管不再吃 FAQ/KB 全表，acknowledge 回應由 Haiku handler 寫。

[用戶訊息] ↓ ┌────────────────────────────────┐ │ phase 攔截 / greeting fast-path │ 純程式 └────────────────────────────────┘ ↓ 沒命中 ┌────────────────────────────────┐ │ pipeline.run（4 個 Haiku 接力） │ ← v5 流水線復活 │ entry_classifier │ │ intent_clarity │ │ faq_matcher → matched_id+conf │ │ kb_indexer → article ids │ │ │ │ 輸出 hint 卡片：分類 / FAQ 命中 │ │ /KB 文章建議 / 偵測意圖 │ └────────────────────────────────┘ ↓ ┌────────────────────────────────┐ │ manager.decide（Sonnet 瘦版） │ │ 看：用戶這句 + history 10 輪 │ │ + intent_log + 流水線 hint │ │ 不再吃 FAQ/KB 全表（流水線權威） │ │ 輸出 12 個 action 選 1 │ └────────────────────────────────┘ ↓ ┌─ executor ─────────────────────┐ │ faq_responder Haiku │ │ cs_response Sonnet (cached) │ │ acknowledge_handler Haiku（v7.2 新） │ │ ... 其他不變 │ └────────────────────────────────┘ ↓ [evaluator Haiku 背景，prompt 已放寬語意]

解的問題（v6 → v7）

  • v6 鬼打牆「OK 了 謝謝」：新增 acknowledge_confirmation action，acknowledge_handler（Haiku）讀 intent_log 自動推進到下一個 pending intent
  • evaluator 漏判：prompt 從「對齊範例」改「看語意不看字面」、新增「包括但不限於」措辭
  • 主管 prompt 過長：移除 FAQ/KB 全表，命中項詳情走 hint 帶進 user prompt（system prompt cache 命中區更小但 input 更短）
  • v5 鬼打牆殘餘：流水線判錯時主管不挑替代 FAQ、改選 acknowledge_uncertainty 請用戶澄清

關鍵設計：流水線當權威 + 拆 acknowledge

  • 流水線權威（v7.1）：主管不再從零挑 FAQ/KB，只能採信流水線命中或選「不走 FAQ/KB」action（acknowledge_*、suggest_ticket、clarify）
  • 拆 acknowledge（v7.2）：主管 Sonnet 仍負責「判定是 confirmation」（決策），acknowledge_handler（Haiku）負責「寫回應」（生成）。各司其職

Session 基本資料

欄位說明
session_id對話 session 唯一 ID（UUID）
created_at / updated_at建立 / 最後更新時間（UTC）
phase對話階段：對話中／等待用戶選擇意圖／等待工單確認／等待 Email／已結束
turn_count對話輪數（greeting 不計）

用戶資料 (user_info)

欄位說明
is_logged_in是否登入會員
user_id / user_name / user_email會員資料（建單時自動帶 email）
purchase_history已購課程清單（mock 資料）

意圖追蹤 (intent_state) ⭐

v6 主管模式的核心：所有偵測到的意圖都會記在 intent_log，永不刪除、只更新狀態。

欄位說明
input_classification本輪 manager 選的 action 名稱
consecutive_unclear_count連續模糊計數，達 3 次強制建單
greeting_countsession 累計 greeting 次數，超過 max（3）轉灰框
current_intent目前正在跟用戶討論的意圖文字
awaiting_selection是否在等用戶從多重意圖選一個
intent_logsession 全部偵測到的意圖陣列（每項含 text / status / role / in_scope / first_turn）

intent_log 內每項的欄位

欄位說明
text意圖文字
statuspending（待辦）／in_progress（討論中）／answered（已回但未確認解決）／confirmed_resolved（用戶明確說解決了）

服務上限機制 (service_limits)

系統有四個累計計數器，達到上限會自動建議建單：

計數器上限觸發條件
off_topic_count3用戶連續 3 次離題
low_confidence_count3AI 對自己答案的信心 < 0.4 累計 3 次
unresolved_count3用戶對 AI 回答不滿意累計 3 次
turn_count20整個 session 對話超過 20 輪

任一條件觸發後 limit_reached 變 true，limit_reached_reason 會記錄是哪個原因， 下一輪用戶說話時系統主動跳出建議建單訊息。

用戶若拒絕建單（回「不用」），user_decision 變 declined，系統不會再重複建議建單； 繼續離題的話會走灰框 off_topic_blocked。

工單流程

  1. 系統建議建單（紫框 ticket_flow）
  2. 用戶回「好」或「不用」（系統用 Haiku 判斷）
    • 好 → 進階段 3
    • 不用 → 結束建單流程，回對話模式（不再重複建議）
    • 看不出意圖 → 退回對話模式，重新交給主管處理
  3. 檢查身分
    • 會員 → 直接用 user_info.user_email 建單
    • 訪客 → 進階段 4 收 Email
  4. 收集 Email（最多重試 3 次格式錯誤後放棄）
  5. Sonnet 生成工單摘要 → 寫入 SQLite tickets 表 → 回工單編號
  6. phase = 已結束，後續輸入固定回「請按新對話按鈕」

後台介面

/admin 提供：

  • 工單列表（依時間倒序）
  • 篩選 Open / In Progress / Closed

內部操作建議

觀察 AI 行為

  • 看左側對話的色帶，可以快速判斷 AI 走的是哪一條路徑
  • 對話下方 response_type 標籤滑鼠移過去會顯示中文說明

測試常見場景

想測什麼輸入範例預期顏色
FAQ 命中「我影片不能看」綠色 faq
RAG 走查「我影片有畫面但沒聲音」藍色 rag
多重意圖「我有發票跟付款問題」深紫 intent_selection
主從關係「我發票要先解決，順便看一下退費」綠／藍（直接處理發票）
離題「你今天午餐吃什麼」橘色 off_topic
系統不知道「你們公司有附設停車場嗎」粉紅 no_kb_match + 建單按鈕
純問候「你好」淺藍虛線 greeting
模糊訊息連發「我有問題」、「不行」、「dlsjfk」第 3 次 → 紫色 force_escalation

有問題時怎麼 debug

  1. 看右側 State 面板：intent_state.input_classification 顯示主管選了哪個 action
  2. 看 intent_log：所有偵測到的意圖、status、role 都在這裡，可看出系統「以為」用戶要做什麼
  3. 看 service_limits：是不是已達某個上限導致系統行為不一樣
  4. 看 ticket_state：用戶是否已拒絕過建單（declined）影響後續邏輯

常見「為什麼 AI 這樣回」的原因

觀察到的行為可能原因
連續被問「我聽不懂」主管判 acknowledge_uncertainty 或 clarify。可能用戶訊息太籠統，或 chat_history 有歷次 clarify 紀錄拖累判斷。
突然冒出「建立工單」按鈕主管選了 suggest_ticket，或服務上限觸發，或 cs_response 加了 [SUGGEST_TICKET] 標記。
第 4 次「你好」變成灰框greeting_count 超過 max（預設 3）後永久轉灰框，不再用 LLM 生問候。
用戶離題系統不擋主管判定屬於 in_scope 範圍（例如帶有「課程」「影片」關鍵詞）。可調主管 prompt 範圍定義。
「下一個問題呢」沒對應到舊意圖主管沒解析到指稱詞，可能 intent_log 已經沒 pending 項或 status 已經 confirmed_resolved。

修改 KB / FAQ
