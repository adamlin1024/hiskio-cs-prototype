這是什麼

HiSKIO AI 客服系統是一個三段式對話客服雛形： 先由 主管（Sonnet） 看完整對話脈絡決定要做什麼動作，再分派給對應的執行者節點（FAQ 回應、KB 查詢、離題擋下、建立工單等）回應用戶。

最終目標：用戶能透過對話得到 FAQ 標準答案、KB 知識庫解答；遇到複雜或非業務範圍的問題會自動建議建立工單由真人客服處理。

內部操作者主要看左側對話、右側即時狀態（State 除錯面板），可隨時了解 AI 為什麼這樣回應、走哪一條路徑。

對話訊息燈號說明

每則 AI 回應左側的色帶代表系統判斷的回應類型，可從顏色快速看出是走哪一條路徑：

淺藍虛線 — 問候 (greeting)純打招呼（你好、Hi、在嗎）。turn_count 不增加。一個 session 累計 max_greeting_count 次（預設 3）後永久轉灰框。
灰色 — 問候已達上限 (greeting_blocked)greeting 累計超過 3 次後的硬擋訊息，提示用戶有問題就直接描述、否則請結束對話。不再呼叫 LLM。
橘黃 — 釐清 (clarification)用戶訊息模糊（亂碼、太籠統、無法判斷想問什麼），系統請用戶補充。連續 3 次強制建單。
橘色 — 離題 (off_topic)用戶問題不在 HiSKIO 業務範圍（食物、天氣、推薦商品、質疑系統等）。off_topic_count 會累加，達上限後觸發建單建議。
灰色（小字）— 離題已達上限 (off_topic_blocked)離題累計達上限後的硬擋訊息：「對話僅處理 HiSKIO 服務相關問題」。
深紫色 — 多重意圖選項 (intent_selection)偵測到多個獨立問題，列選項請用戶選一個先處理。
綠色 — FAQ 標準答案 (faq)命中 FAQ，AI 用核心步驟一字不漏照抄、潤飾開場結尾。穩定、不會幻覺。
藍色 — 知識庫解答 (rag)FAQ 未命中、KB 有相關文章，由 Sonnet 根據 KB 內容生成回答。
粉紅色 — 知識庫無解 (no_kb_match)KB 完全沒有相關文章，AI 誠實告知並建議建立工單。
紫色 — 工單流程 (ticket_flow / force_escalation)建議建單／確認建單／收 Email／建單成功 等工單相關訊息。force_escalation 是「連續 3 次無法溝通」的強制建單。
草綠 — 確認語回應 (acknowledge) ⭐ v7 新增用戶說「謝謝」「我知道了」「OK」等確認語，主管直接回禮貌過渡語並把當前 intent 標 confirmed_resolved，不再走 KB（解 v6 鬼打牆）。
紅色 — 對話結束 (session_ended)工單已建立或對話結束，提示用戶按「新對話」重新開始。

系統運作流程

每一輪用戶訊息進來，依以下順序處理（前一步沒命中才會進到下一步）。每個步驟有標明用什麼模型（Sonnet/Haiku）或純程式（regex）：

  1. ① 特殊 phase 攔截 純程式
    檢查 state.phase：如果在等待工單確認、等待 Email、等待用戶從多重意圖選一個、對話已結束，直接交給對應 handler 處理。不呼叫 LLM。
  2. ② Greeting fast-path regex 純程式
    regex 比對「整句剛好是純招呼字串」（你好／Hi／嗨／在嗎／hello）。
    命中 → 跳過 Sonnet 主管，直接由 Haiku 生問候語回應
    沒命中（例如「helloo」「你好啊請問」）→ 往下走主管判斷
  3. ③ Sonnet 主管統一決策 Sonnet 4.6
    看「完整對話歷史 + intent_log + 系統狀態 + FAQ 清單 + KB 索引」，輸出一個 action（共 10 種）+ 該 action 需要的 payload（要用哪個 FAQ、哪些 KB 文章等）。
    有 prompt caching：固定部分（規則 + FAQ list + KB 索引）走快取省 90% input cost。
  4. ④ 執行者派任務 純程式
    orchestrator 看主管 action 派給對應節點。每個節點用的模型不同：
    • faq_responder → Haiku（照 FAQ 模板潤飾）
    • cs_response → Sonnet（用 KB 生 RAG 答案，有 caching）
    • no_kb_handler → Haiku（誠實告知不知道）
    • off_topic → Haiku（離題回應）
    • greeting_handler → Haiku（生問候語）
    • clarification_handler → Haiku（請用戶補充）
    • ticket_handler.decide → Haiku（判 yes/no）
    • ticket.generate_summary → Sonnet（生工單摘要）
  5. ⑤ 背景評估 Haiku 4.5
    對話結束後跑評估，把情緒、分類、是否確認解決等資訊寫回 State 供下輪參考。
常見誤解澄清：「greeting 是不是用 Sonnet 處理？」
  • 純招呼（regex 命中）→ 完全不打 Sonnet，由 Haiku 直接生問候
  • 帶錯字／變化型招呼（regex 沒命中）→ Sonnet 主管接手判斷後也選 greeting action → 還是叫 Haiku 生問候
  • 無論哪條路徑，最終問候訊息都是 Haiku 生成的。差別只在「有沒有呼叫 Sonnet 做路由判斷」（影響成本）

下表列出主管所有可選 action。實際各 action 的觸發頻率有差，例如 greeting 因為已被 fast-path 攔截，主管很少走到。

list_pending_intents用戶用「下一個」這類指稱詞，列出待辦意圖多重意圖場景

服務上限機制 (service_limits)

系統有四個累計計數器，達到上限會自動建議建單：

計數器上限觸發條件
off_topic_count3用戶連續 3 次離題
low_confidence_count3AI 對自己答案的信心 < 0.4 累計 3 次
unresolved_count3用戶對 AI 回答不滿意累計 3 次
turn_count20整個 session 對話超過 20 輪

任一條件觸發後 limit_reached 變 true，limit_reached_reason 會記錄是哪個原因， 下一輪用戶說話時系統主動跳出建議建單訊息。

用戶若拒絕建單（回「不用」），user_decision 變 declined，系統不會再重複建議建單； 繼續離題的話會走灰框 off_topic_blocked。

工單流程

  1. 系統建議建單（紫框 ticket_flow）
  2. 用戶回「好」或「不用」（系統用 Haiku 判斷）
    • 好 → 進階段 3
    • 不用 → 結束建單流程，回對話模式（不再重複建議）
    • 看不出意圖 → 退回對話模式，重新交給主管處理
  3. 檢查身分
    • 會員 → 直接用 user_info.user_email 建單
    • 訪客 → 進階段 4 收 Email
  4. 收集 Email（最多重試 3 次格式錯誤後放棄）
  5. Sonnet 生成工單摘要 → 寫入 SQLite tickets 表 → 回工單編號
  6. phase = 已結束，後續輸入固定回「請按新對話按鈕」

後台介面

/admin 提供：

  • 工單列表（依時間倒序）
  • 篩選 Open / In Progress / Closed
  • 單筆工單可標記狀態（標記 Open / 處理中 / 結案）
  • 查看完整對話 modal：含 AI 摘要 + AI 嘗試過什麼 + 完整 chat_history（每則含 response_type）

內部操作建議

觀察 AI 行為

  • 看左側對話的色帶，可以快速判斷 AI 走的是哪一條路徑
  • 對話下方 response_type 標籤滑鼠移過去會顯示中文說明
  • 右側 State 面板可即時看到所有狀態變化（特別注意 intent_log 與 service_limits）

測試常見場景

想測什麼輸入範例預期顏色
FAQ 命中「我影片不能看」綠色 faq
RAG 走查「我影片有畫面但沒聲音」藍色 rag
多重意圖「我有發票跟付款問題」深紫 intent_selection
主從關係「我發票要先解決，順便看一下退費」綠／藍（直接處理發票）
離題「你今天午餐吃什麼」橘色 off_topic
系統不知道「你們公司有附設停車場嗎」粉紅 no_kb_match + 建單按鈕
純問候「你好」淺藍虛線 greeting
模糊訊息連發「我有問題」、「不行」、「dlsjfk」第 3 次 → 紫色 force_escalation

有問題時怎麼 debug

  1. 看右側 State 面板：intent_state.input_classification 顯示主管選了哪個 action
  2. 看 intent_log：所有偵測到的意圖、status、role 都在這裡，可看出系統「以為」用戶要做什麼
  3. 看 service_limits：是不是已達某個上限導致系統行為不一樣
  4. 看 ticket_state：用戶是否已拒絕過建單（declined）影響後續邏輯

常見「為什麼 AI 這樣回」的原因

觀察到的行為可能原因
連續被問「我聽不懂」主管判 acknowledge_uncertainty 或 clarify。可能用戶訊息太籠統，或 chat_history 有歷次 clarify 紀錄拖累判斷。
突然冒出「建立工單」按鈕主管選了 suggest_ticket，或服務上限觸發，或 cs_response 加了 [SUGGEST_TICKET] 標記。
第 4 次「你好」變成灰框greeting_count 超過 max（預設 3）後永久轉灰框，不再用 LLM 生問候。
用戶離題系統不擋主管判定屬於 in_scope 範圍（例如帶有「課程」「影片」關鍵詞）。可調主管 prompt 範圍定義。
「下一個問題呢」沒對應到舊意圖主管沒解析到指稱詞，可能 intent_log 已經沒 pending 項或 status 已經 confirmed_resolved。

修改 KB / FAQ

  • FAQ：直接編輯 data/faq.json，重啟 server 即生效
  • KB：把 .md 放到 data/kb/ → 跑 python scripts/build_kb_index.py → 重啟 server

版本演化（v5 / v6 / v7）

這個雛形經歷三個架構版本。共用基礎建設下方先列、再分版本說明各自做了什麼、解了什麼問題、踩了什麼坑。

共用基礎（三版都不變）

面向內容
State 結構session_id / phase / turn_count / user_info / issue_context / faq_context / kb_context / service_limits / ticket_state / intent_state / escalation_signals / chat_history
service_limitsmax_turns_per_session=20、max_off_topic_count=3、max_low_confidence_count=3、max_unresolved_count=3
phase 攔截等待用戶選擇意圖 / 等待工單確認 / 等待 Email / 已結束 — 在 manager 之前優先處理
greeting fast-pathregex 攔下純問候，不打 LLM。greeting_count 上限 3，超過鎖灰框
會員/訪客識別會員從 mock_users.json 抓 email；訪客在工單流程收 Email（regex 驗證）
FAQ 回應規則核心步驟一字不漏照抄，由 faq_responder 只潤飾開頭結尾（任何版本都不准 AI 改 core_steps）
KB 兩層分離data/kb_source/（原稿）→ data/kb/（系統檔）→ kb_index.json（索引），由 build_kb_index.py 產出
evaluator 職責每輪 AI 回應後背景跑 Haiku，更新 issue_context、low_confidence_count、unresolved_count、intent_log status（confirmed_resolved）
工單流程建議 → 確認 → 收 Email → Sonnet 生工單摘要 JSON → 寫進 SQLite tickets 表

v5 — 分類器流水線（已歸檔，branch=v5-pipeline-mode）

架構：4 個 Haiku 接力跑 → entry_classifier → intent_clarity → faq_matcher → kb_indexer → cs_response/faq_responder。每節點看自己負責的一小塊資料，輸出傳給下一個。

解的問題：相比更早的 v3、把 router 拆成更精細的入口分類（greeting / unclear / off_topic / customer_service）+ 意圖明確度（simple / parallel_multiple / 指稱詞）+ 平行多意圖追蹤。

踩到的 bug

bug實況
「人工建單為什麼以後不能動作了」鬼打牆entry_classifier 看到「人工建單」字眼歸 customer_service → kb_indexer 被歷史污染挑 kb_005（發票）→ cs_response 拿到 KB 全文回「請問您的發票問題是哪方面...」。沒有任何節點看出用戶在質疑系統。
同樣意思走不同 path 結果不一致選號「3」走 phase 攔截、陳述句「我有水果問題」走 entry_classifier，兩條路結果不一樣。
ticket_state.declined 永久封印用戶拒絕一次工單後、之後新意圖也不再被建議建單。
邊界 case 永遠補不完新訊號（meta 質疑、「我聽不懂」、混合 in/out scope）要在 entry_classifier、faq_matcher、kb_indexer、cs_response 都加分支。

v6 — 主管統一決策（main 上一版）

架構變動：拆掉 4 個 Haiku 分類器、改由一個 Sonnet 主管看「全貌」一次決策。主管輸出 11 個 action 之一，分派給 executor 節點生回應。

解的問題

  • v5 鬼打牆：主管看完整 history + intent_log + FAQ 全表 + KB 索引，能識別「meta 質疑」並選 suggest_ticket
  • 同意思走不同 path：全部進主管，只有一條 code path
  • 「我聽不懂」一級公民：新增 acknowledge_uncertainty action（誠實出口、不再硬塞分類）
  • 邊界 case 加 action enum 一處改完，不必到處改節點

實測成本（10 輪 benchmark）：開 Sonnet prompt cache 後 NT$3.81 / 10 輪。

v6 也踩到的新 bug

bug實況
「我知道了」鬼打牆 ⭐用戶說「好 我知道了」「OK 了 謝謝」確認語時，主管的 11 個 action 沒有對應出口、硬塞 continue_intent → 把當前 intent 主題又重答一遍。
evaluator 漏判 confirmed_resolvedevaluator prompt 規則「對齊 5 個範例」（問題解決了 / OK 了 / 謝謝 / 沒問題了 / 搞定）。Haiku 在 t=0 嚴格字面對齊，「我知道了 / 了解了 / 明白了」這類常見表達被判 false。
主管 prompt 越加越笨邊界 case 都壓在 system prompt 裡（FAQ 全表 + KB 索引 + 規則）越加越長，Sonnet 注意力分散、混合意圖（in_scope + out_of_scope）容易判 clarify。

v7 — 流水線 + 主管混合制（current main）

架構變動：保留 v6 主管統一決策，但加回 v5 流水線當「預判」：

  1. 用戶訊息進來
  2. phase 攔截 / greeting fast-path 不變
  3. 新增：跑 4 個 Haiku 流水線（entry_classifier / intent_clarity / faq_matcher / kb_indexer），把結果濃縮成「提議卡」
  4. 主管 Sonnet 看用戶這句 + 提議卡 + 完整 history + intent_log → 從 12 個 action 選 1（多了 acknowledge_confirmation）
  5. 主管可採信流水線提議、也可 override（流水線只是參考、不是命令）
  6. executor 執行 action
  7. evaluator 兜底（prompt 已放寬語意）

解的問題

  • 「我知道了」鬼打牆：新增 acknowledge_confirmation action，主管直接寫禮貌回應 + 標 intent confirmed_resolved，不走 KB
  • evaluator 漏判：prompt 從「對齊範例」改「看語意不看字面」，新增「包括但不限於」措辭
  • 主管多了流水線「眼線」，KB 污染 / 質疑語氣這類 case 主管能看 hint 信心值再 override

  • v7（初版，主管 system prompt 仍含 FAQ/KB 全表）：NT$4.39 / 10 輪 — 比 v6 貴 15%
  • v7.1（current main）：NT$3.74 / 10 輪 — 比 v6 還省 2%。主管 system prompt 移除 FAQ/KB 全表（流水線權威）、命中項詳情走 hint 帶進 user prompt

v7.1 關鍵設計：流水線當權威。主管不再從零挑 FAQ/KB，只能採信流水線命中、或選「不走 FAQ/KB」的 action（acknowledge_*、suggest_ticket、clarify）。流水線挑錯時主管會選 acknowledge_uncertainty 請用戶澄清，不會硬挑替代 FAQ。
